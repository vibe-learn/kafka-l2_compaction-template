        # kafka — Log compaction vs retention by time

        Homework-шаблон для урока **l2_compaction** (Log compaction vs retention by time) на платформе Vibe Learn.

        ## Что делать

        Реализуй Go-программу для демонстрации log compaction в Kafka. Требования:
- Создай compact-топик `user-state` с `cleanup.policy=compact` и `min.cleanable.dirty.ratio=0.1`.
- Продьюсер отправляет 1000 сообщений с ровно 10 уникальными ключами (`key-0` .. `key-9`),
  каждый ключ обновляется ~100 раз с разными значениями.
- После отправки всех сообщений вызови явный триггер compaction (через конфиг или ожидание).
- Consumer читает топик от начала (`auto.offset.reset=earliest`) и собирает финальные значения.
- CI-проверка (assert): итоговое количество уникальных ключей = 10; для каждого ключа
  значение соответствует последнему отправленному продьюсером.
Тесты в template-репо проверят: корректность конфигурации топика, поведение tombstone
(отправь tombstone для `key-5`, убедись что он исчезает после compaction).

## Контекст (из transfer-задачи урока)

Команда разработки хранит текущий баланс пользователей в Kafka-топике `user_balance`.
Ключ — `user_id` (строка), значение — текущий баланс в копейках (int64).
За месяц один активный пользователь совершает ~10 миллионов мелких транзакций.

Аналитик предлагает: «Давайте просто поставим `cleanup.policy=delete` с `retention.ms=90days` —
будем хранить все обновления». Инженер возражает: «Нет, надо `compact` — тогда место не кончится».

## Recap из урока

- `cleanup.policy=compact` хранит только **последнее значение per-key** — история изменений безвозвратно теряется при каждом новом write для существующего ключа.
- Compaction подходит для **state stores, config-топиков, CDC-снапшотов** — где важно текущее состояние. Не подходит для event log, audit log, финансовых транзакций.
- **Tombstone** (запись с `value=null`) — единственный способ явно удалить ключ из compact-топика. После `delete.retention.ms` ключ исчезает полностью.
- `cleanup.policy=delete,compact` даёт обе семантики: уплотнение per-key **и** ограничение retention по времени/размеру — полезно для state stores с конечным сроком жизни.
- `min.cleanable.dirty.ratio` управляет агрессивностью cleaner-потока: ниже значение → чаще compaction → чище лог, но выше I/O-нагрузка на брокер.

        ## Как работать

        1. Платформа Vibe Learn создаёт копию этого репо в твоём GitHub-аккаунте по клику «Начать домашку» на странице урока (через GitHub `/generate`, codecrafters-pattern).
        2. Склонируй копию локально, реализуй TODO в `main.go`, прогони тесты, запушь.
        3. CI (`.github/workflows/ci.yml`) запускает `go vet` + `go test ./...` на каждый push. Платформа слушает результат через webhook от GitHub Actions и обновляет статус домашки на странице урока.

        ## Локальное окружение

        - Go 1.22+
        - Docker + docker-compose — `docker compose -f docker-compose.yml up -d` поднимает 3-нодовый Kafka cluster на портах 9092/9093/9094, использовать в тестах через bootstrap `localhost:9092,localhost:9093,localhost:9094`.

        ## Запуск

        ```bash
        # Поднять локальный Kafka
        docker compose up -d

        # Прогнать тесты (часть из них стартует свой ephemeral testcontainers cluster, часть использует docker-compose выше)
        go test ./...

        # Запустить main (печатает marker; замени stub на реализацию)
        go run .
        ```
